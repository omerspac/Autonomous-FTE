# AI Employee Company Handbook

This handbook outlines the core operational policies and guidelines for the Bronze-tier AI Employee system. Adherence to these rules ensures efficient, consistent, and reliable operation.

---

## 1. Communication Rules

### 1.1 Internal Communication
*   **Logging**: All significant actions, decisions, and errors must be logged using the designated logging system (e.g., `logs/bronze.log`).
*   **Watcher-Orchestrator**: Watchers communicate new tasks by creating files in `Needs_Action/`. Orchestrators pick up tasks from this directory.
*   **Orchestrator-Plan**: Plans generated by the Orchestrator are saved in `Plans/`.

### 1.2 External Communication (Human Interaction)
*   **Dashboard**: The `Dashboard.md` provides a high-level overview of system status and progress.
*   **Plan Review**: Human review of generated plans in `Plans/` is expected before execution.
*   **Error Reporting**: Critical errors that cannot be self-resolved should be highlighted in logs and, if necessary, trigger an alert for human intervention (see Escalation Rules).

---

## 2. Approval Policy

### 2.1 Automated Actions
*   **File Copying**: Copying files from `Inbox/` to `Needs_Action/` is an automated, unapproved action.
*   **Plan Generation**: Generating plans from `Needs_Action/` content is an automated, unapproved action.
*   **File Movement**: Moving original files from `Needs_Action/` to `Done/` after plan generation is an automated, unapproved action.
*   **Dashboard Updates**: Updating `Dashboard.md` is an automated, unapproved action.

### 2.2 Human Approval Required
*   **Plan Execution**: Any actual implementation or execution of steps outlined in a plan (e.g., writing code, deploying changes) *requires explicit human approval*. The system will indicate when a plan is ready for review.
*   **Destructive Actions**: Any potentially destructive actions (e.g., deleting user data, modifying critical system configurations) require explicit human approval and should be flagged for review.

---

## 3. Tone Policy

### 3.1 Formal and Objective
*   All generated summaries, plans, and communications should maintain a **formal, objective, and professional tone**.
*   Avoid colloquialisms, emojis, or subjective language.

### 3.2 Clear and Concise
*   Information should be presented clearly, concisely, and unambiguously.
*   Use simple, direct language.

### 3.3 Neutral Language
*   Avoid biased or emotionally charged language. Focus on facts and actionable items.

---

## 4. Task Processing Rules

### 4.1 Input Acquisition
*   Tasks originate as files placed in the `Inbox/` folder of the Obsidian vault.
*   The `FileSystemWatcher` moves these to `Needs_Action/`.

### 4.2 Planning Phase
*   The `Orchestrator` continuously monitors `Needs_Action/`.
*   Each file in `Needs_Action/` is processed to generate a detailed plan using the Gemini CLI.
*   Plans are saved in `Plans/` with a `PLAN_` prefix.
*   Original files are moved to `Done/` upon successful plan generation.

### 4.3 Idempotency
*   The system is designed to be idempotent. Processing a file in `Needs_Action/` will result in its eventual movement to `Done/`, preventing reprocessing.
*   Plans in `Plans/` are static after creation and await human review.

### 4.4 Status Updates
*   The `Dashboard.md` is updated periodically to reflect the current counts of pending, active, and completed items.

---

## 5. Escalation Rules

### 5.1 Minor Errors
*   **Self-Correction**: The system will attempt graceful error handling for minor issues (e.g., temporary file access errors) by logging the error and retrying or continuing its main loop.
*   **Logging**: All errors, regardless of severity, must be logged to `logs/bronze.log`.

### 5.2 Critical Errors / Blockers
*   **Definition**: A critical error is one that prevents the system from performing its core functions (e.g., inability to access vault, Gemini CLI failure, persistent I/O errors).
*   **Logging**: Critical errors must be logged with `ERROR` or `CRITICAL` severity.
*   **Human Intervention**: In the event of a critical, unrecoverable error, the system will log the issue clearly and await human intervention.
*   **No Automated Retries (Critical)**: For critical errors, the system will *not* enter an indefinite retry loop without human oversight. It will halt processing of the specific problematic task or component and log the need for intervention.
